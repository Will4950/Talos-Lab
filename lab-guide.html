01Architecture

Roles split, hosts stacked

Every Proxmox host runs two Talos VMs: a control-plane node and a worker node. Control planes handle etcd and the API; workers carry every workload and all the storage - each on its own dedicated data disk.

Keeping the roles on separate VMs means etcd never competes with a busy pod for CPU or disk, and you can reboot or rebuild a worker without touching the control plane. The tradeoff is a couple more VMs - cheap on a homelab. The layout grows by the host: add a Proxmox box, add one control-plane node and one worker node, and the cluster absorbs them.

The honest bit about HA at two hosts

Real control-plane fault tolerance needs three separate hosts. With two hosts you run two control planes, and a two-member etcd can't hold quorum if one goes away - lose a host and your workloads keep running, but the cluster goes read-only until it returns. This is a property of two-host clusters, not of this setup. The third host is where quorum becomes fault-tolerant, so the config is built to make adding it trivial.

Talos itself is an immutable, API-driven Linux with no SSH and no shell - you never log into a node, you send it declarative machine config. Argo CD then manages everything inside the cluster from Git, and Sealed Secrets lets even your secrets live safely in that repo. The whole lab is reproducible from one config file and a directory of small scripts.

03Prerequisites

Before you start

This builds a cluster from scratch; it does not attach to an existing one. Nothing here needs to exist yet except the ground it stands on: the hosts, a working network with DHCP, outbound internet, and a few CLI tools on your machine. Get the pieces below in place first and the phases in §9 run start to finish.

Get the scripts: github.com/Will4950/Talos-Lab

Hardware

  • Two or more Proxmox VE hosts (9.x) on the same L2 network; add as many as you like. You'll run two VMs on each.
  • Size per role: control planes stay small (etcd + the API), workers get more (they carry the pods and all the storage). Defaults are 2 vCPU / 4 GB for control planes and 4 vCPU / 8 GB for workers, on a 40 GB OS disk - all tunable, and overridable per node when one host differs.
  • Each worker also needs a second, blank disk for Longhorn (100 GB by default). The scripts create it when they provision VMs; if you build the VMs yourself, add a blank disk to each worker.

Your workstation

The bootstrap runs from your machine, not the hosts. It reads the YAML config with yq and seals secrets with kubeseal.

One note: kubeseal should match the controller version you set in the config (0.38.4), so its sealed format lines up with what the in-cluster controller understands. Either yq works - the Go build from mikefarah (v4) or the Python/jq-based one; the scripts stick to filters both understand.

Network & what must be reachable

Most first-run trouble is network, not the scripts. These need to hold before you start:

  • All hosts on one L2 segment / subnet. With more than one host, the bridge or SDN VNet must flood ARP across hosts - the API VIP (talos-vip) and MetalLB both announce over ARP, so a host-local zone breaks cross-host traffic (see §16).
  • A DHCP server on that subnet that supports reservations (MAC to a fixed IP). Your node IPs sit in the subnet but outside the DHCP pool, and the MetalLB pools stay clear of it too.
  • Outbound internet from the nodes. On first boot they pull the Talos image from factory.talos.dev and, once up, container images and Helm charts from public registries.
  • Your workstation can reach the node IPs and the API endpoint over the LAN - Talos API on 50000, Kubernetes API on 6443.
  • Host clocks roughly in sync (NTP). etcd is time-sensitive and a large skew stops nodes joining.

Optional, only if you use them: root SSH to each host (lets the scripts build the VMs and prep the bridge); a private Git repo plus a deploy key for the Argo CD app-of-apps (§10); an OPNsense box if you want the scripts to write DHCP reservations for you (§9, step 0).

04Network plan

Addressing on 192.168.4.0/24

Your DHCP server hands out 192.168.4.110–220, so every static address here - nodes and the API endpoint - sits below it, and MetalLB takes the free blocks on either side. Nothing the cluster owns overlaps the DHCP range.

PurposeAddressNotes
Gateway192.168.4.254Your existing router
Proxmox hosts192.168.4.5–.6pve1, pve2 (expand .7, .8)
API VIP192.168.4.10Shared control-plane endpoint the nodes float (talos-vip) - a free IP, not a reservation
Control planes192.168.4.11–.12One per host (expand .13, .14)
Workers192.168.4.21–.22One per host (expand .23, .24)
DHCP pool192.168.4.110–.220Off-limits - leave entirely to DHCP

MetalLB (high)192.168.4.221–.250App IPs above the DHCP range

That's roughly 108 addresses for LoadBalancer services across the two MetalLB ranges - plenty of headroom, and easy to extend with more ranges or named pools later.

Reserve before boot

Create the control-plane and worker DHCP reservations before the VMs first boot (or reboot them after). A node that boots without its reservation lands somewhere in .110–.220 and the bootstrap won't find it where it expects. On OPNsense (dnsmasq) the dhcp phase can create them for you (GUI-visible, via the API) - see §6 and §9.

05Repo layout

Where everything lives

Two directories carry the whole lab: scripts/ holds the bootstrap logic, and secrets/ holds your configuration and the generated artifacts. You author exactly one file - secrets/cluster.yaml.

secrets/ stays out of git

The shipped .gitignore excludes secrets/cluster.yaml and secrets/out/ - they hold repo tokens, key paths, and the generated machine configs, which contain your cluster's certificates and secrets. Only cluster.example.yaml is tracked.

06Build the config

Write secrets/cluster.yaml, block by block

Start from the template, then build it up a section at a time. Each block below explains a decision you're making - by the end you'll understand every line, not just have pasted one.

1Cluster identity and versions

Name the cluster and pin every component. Leaving kubernetes blank uses the version that ships with your Talos release (1.36), which is what you want unless you're deliberately skewing them.

2Network

One flat subnet. endpointMode sets how the stable API address (apiEndpoint) is served. The default, talos-vip, is fully self-hosted: the control planes elect an owner via etcd and that node answers for 192.168.4.10 - nothing outside the cluster is involved. In-cluster components don't even use it (they reach the API through KubePrism); the VIP is just the stable address for kubectl/talosctl and the nodes. apiEndpoint must be a free IP outside the DHCP range. Leave kubeSpan off unless your hosts live on different subnets or sites.

Make the VIP reach across your Proxmox hosts

The owner answers ARP for 192.168.4.10 on the L2 segment, so it only reaches other hosts if the segment carries that ARP. On Proxmox SDN two things trip this up - and both live in Proxmox, not in some external box:

  1. Turn the Proxmox firewall off on the control-plane NICs. With it on, PVE's IP/ARP filter drops ARP for any address that isn't the VM's assigned IP - which is exactly the VIP. This suite provisions NICs with firewall=0; for VMs you already created, clear it (keep your MAC): qm set <vmid> --net0 virtio,bridge=lab,firewall=0.
  2. The VNet must flood broadcast/ARP between hosts. A VLAN zone does this natively (if the VLAN is trunked on the uplink); a VXLAN zone does when BUM/head-end replication is on (the Proxmox default). A Simple zone is host-local and can't carry a VIP across hosts at all; an EVPN zone turns on ARP/ND suppression by default, so the VTEP answers ARP from its BGP-learned cache instead of flooding a floating VIP's gratuitous ARP - the usual reason a VIP is invisible across hosts on EVPN. Fix it in Proxmox: Datacenter → SDN → Zones → your EVPN zone → tick Disable ARP-ND Suppression, then Apply (nothing external). Advertising the VIP over BGP is the alternative if you'd rather keep suppression on.

Prove the network path in two minutes without touching Talos: create two throwaway Linux VMs or LXCs on the lab VNet, one on each host, then have one impersonate the VIP:

Replies on box B mean the VNet carries the VIP's ARP across hosts, so the Talos VIP will work - no answer means it doesn't, and the fix is one of the two points above (firewall off, and for EVPN the suppression toggle). To confirm the real thing later, bring one control plane up and run this from a node on a different host:

Answers from the other host mean the VIP is live cluster-wide, with nothing external. No answer means ARP isn't crossing the VNet - re-check the two points above. If the zone still can't carry the VIP's ARP across hosts (a Simple zone, or EVPN with suppression left on), move that VNet to a VLAN or VXLAN zone - both flood ARP natively, and on a VLAN (which is what this lab assumes) the VIP just works. Swapping VIP software such as kube-vip does not help, since it rides the same ARP path.

3Install target and the role switch

The disk is where Talos installs on each VM - on a Proxmox VirtIO-SCSI VM that's usually /dev/sda. The important line is the second one: scheduleOnControlPlane: false keeps the control planes tainted, so every pod and every Longhorn replica lands on the workers. That single flag is what makes this a role-separated cluster instead of a hyperconverged one.

4Nodes - one control plane + one worker per host

This is the heart of the file. Each entry names a node's IP and its host (the Proxmox box it lives on, which becomes its failure-domain label). The vmid and mac are only used if you let the scripts build the VMs. To grow the cluster, you uncomment one control plane and one worker per new host - nothing else changes. Any entry can also set cores, memory, or diskGB to override the per-role sizing (step 7) for that one node.

Why one per host, not three control planes on two boxes

Stacking a third control plane onto one of two hosts doesn't buy real HA - that host is still a single point of failure. The clean rule is one control plane per host; quorum becomes fault-tolerant naturally when you reach three hosts. Until then you have a working two-control-plane cluster and a one-line path to three.

5Longhorn and its dedicated disk

The replica count is a Longhorn setting, not a cluster.yaml one: set defaultReplicaCount and defaultClassReplicaCount in manifests/longhorn/values.yaml (Argo CD applies it), at or below your worker count minus any excluded hosts (see below). Because each host has exactly one worker, Longhorn's default node anti-affinity already puts replicas on different hosts, so you don't need the hard zone rule. What you set here is the storage decision - the dedicatedDisk block: each worker gets a second, blank disk that Talos mounts at /var/mnt/longhorn, and Longhorn stores its replicas there, keeping storage I/O and capacity entirely off the OS disk.

If one host has a smaller or overcommitted Proxmox thin pool, use excludeHosts to skip Longhorn storage on that host's workers - they run workloads but host no replicas, and get no data disk (saving pool space). Any worker can also override the default data disk size with dataDiskGB in the node definition (under nodes.workers), useful when one host needs a smaller disk than the rest.

How the disk gets mounted

When enabled: true, 03-gen-config.sh adds a Talos UserVolumeConfig to the worker config that claims the blank disk matched by diskSelector and mounts it at /var/mnt/longhorn. Talos requires a size bound on the volume, so minSize is set and grow: true expands it to fill the disk (keep minSize at or below the data-disk size). The disk must be empty - a disk with an existing filesystem won't match. If you set scheduleOnControlPlane: true, control planes get the disk and mount too.

Exclusion: Workers on hosts in excludeHosts get no data disk (the provisioning script skips --scsi1 for them), no UserVolumeConfig, and 06-kubeconfig.sh never labels them with node.longhorn.io/create-default-disk - so Longhorn never auto-creates a disk there and replicas never land on that node.

6MetalLB - the app address pools

Here's where the DHCP range matters. Since .110–.220 belongs to DHCP, MetalLB gets the two free blocks around it. A pool can list several ranges, and you can add more named pools later (say, to separate internal from external addresses). This is your app IPAM in five lines.

7Proxmox - only if the scripts build your VMs

Leave provisionVMs: false if you create the VMs yourself (by hand or Terraform). Set it true and fill in hosts to have 02-provision-vms.sh create and start them over SSH - it maps each node's host to an ssh target here, and gives each worker the second data disk.

Right-sizing by role and by host

controlPlane and worker set the CPU/RAM default for each role; a node in the nodes block can override cores, memory, or diskGB when one Proxmox host has more (or less) to give. These sizes apply only when the scripts build the VMs - if you create the VMs yourself, set the sizes there.

8Argo CD, Sealed Secrets, and timeouts

Optional: point Argo CD at a private repo of Application manifests and it'll sync your whole platform after bootstrap (you wire up the key in §10). sealedSecrets installs the controller that lets you keep encrypted secrets in that same repo (§11). The timeouts cover how long the scripts wait for health and for each node's maintenance API.

9OPNsense DHCP reservations (optional)

If your DHCP server is an OPNsense box running dnsmasq, the dhcp phase can create the nodes' reservations for you through the OPNsense API - so they show up under Services → Dnsmasq DNS & DHCP → Hosts, editable like any other entry. Fill in the opnsense block; leave enabled: false to skip it. Each node's hostname is its optional name field, else <prefix>-<role>-<n> (e.g. talos-cp-1). domain is optional - leave it blank for a plain DHCP reservation (no DNS host override); dnsmasq still registers the name under the DHCP range's own domain when the node leases. Matching is by MAC, so re-running updates entries in place even if you change the name or domain.

One-time setup on OPNsense

Four things need to be true:

  • dnsmasq is your DHCP server - OPNsense 25.1+ with dnsmasq enabled and serving DHCP (not ISC/KEA). The Hosts API only exists there.
  • An API credential - System → Access → Users → (edit your user) → API keys → + downloads a file with key= and secret=; put those in the block above.
  • That user may access Dnsmasq - its group needs the Services: Dnsmasq DNS/DHCP privilege (an admin account already has it). Without it the API returns 403.
  • A DHCP range on the nodes' subnet - the node IPs may sit outside the pool as long as they're in that same subnet. baseURL is just scheme + host (+ port), no trailing path.
That's the whole file

Those blocks are the complete configuration. The shipped cluster.example.yaml is exactly this with fuller inline comments - see §17 for the whole thing in one piece.

08What the scripts handle

Three Talos details you don't have to remember

Longhorn and MetalLB need capabilities a locked-down OS doesn't grant by default. These are the things that trip up hand-rolled Talos clusters - the scripts bake them in so your cluster.yaml stays high-level.

1 · System extensions in the image

Talos ships no iSCSI tooling and no nsenter, and you can't modprobe them onto an immutable image. 01-schematic.sh layers them in at build time as official extensions, producing a custom installer:

These kernel args are baked into the boot image. On UEFI - which the OVMF VMs use - Talos boots a UKI (systemd-boot) whose command line comes from this schematic, so the serial console covers both the maintenance/ISO boot and the installed system. On that path machine.install.extraKernelArgs is ignored - and newer Talos rejects it - so kernel args belong here in the schematic, not the install section.

2 · The dedicated disk and its kubelet mount

On the workers, 03-gen-config.sh adds a Talos UserVolumeConfig that claims the blank data disk and mounts it at /var/mnt/longhorn - plus a matching kubelet bind mount, because Talos runs the kubelet in its own mount namespace. Longhorn's data path (set in the install) points at the same place.

With the dedicated disk turned off, the same mount is applied at /var/lib/longhorn on the ephemeral partition instead - simpler, but storage then shares the OS disk.

3 · Privileged namespaces

Talos enforces a baseline Pod Security profile on every namespace except kube-system. Longhorn's engine pods and MetalLB's speaker need more than baseline allows, so their namespaces are labelled pod-security.kubernetes.io/enforce: privileged - for Longhorn through Argo CD's managed-namespace metadata, for MetalLB directly. Sealed Secrets installs into kube-system, which is exempt, so it needs no such label.

Verify after boot

Confirm the extensions and the data disk landed with talosctl -n 192.168.4.21 get extensions and talosctl -n 192.168.4.21 get volumestatus on a worker.

11Sealed Secrets

Encrypt secrets so they can live in Git

The sealed-secrets phase installs Bitnami's controller into kube-system as sealed-secrets-controller - the exact name and namespace the kubeseal CLI looks for by default. The controller holds a private key; you encrypt a normal Kubernetes Secret into a SealedSecret that's safe to commit, and only the in-cluster controller can decrypt it. Nobody with repo access can read the secret.

14Persistent storage

Longhorn on the dedicated disk

Longhorn comes up as the default StorageClass, running on workers that host storage (all workers except any on hosts in excludeHosts), with its data path set to /var/mnt/longhorn - the dedicated disk each storage worker got at install. A PVC with no class named just works: it gets replicas placed one per host's data disk, isolated from the OS disk's I/O.

Replicas vs. storage nodes

Node anti-affinity is a hard rule, so a volume can't place more replicas than you have storage nodes. Keep defaultReplicaCount (and defaultClassReplicaCount) in manifests/longhorn/values.yaml at or below your storage node count (worker count minus any excluded hosts), and raise it as you add storage nodes (two storage nodes → 2, three → 3). If you use excludeHosts, those workers don't count toward the storage node total.

Longhorn can back up volumes to an S3-compatible target (MinIO, AWS S3), but this lab doesn't configure one - there's no backupTarget in manifests/longhorn/values.yaml. Enabling it (a target URL, credentials, and a recurring backup job) is an extra step; see the Longhorn backup docs. Automate the upgrade - the update phase

Rather than bumping versions and rolling nodes by hand, scripts/98-update.sh does the whole thing. It is deliberately not in the default ./bootstrap.sh pipeline because it reboots every node in sequence. It fetches the latest upstream releases, rewrites secrets/cluster.yaml (Talos, Argo CD, MetalLB, Sealed Secrets) and the four chart apps/*.yaml (Pi-hole, Longhorn, csi-driver-nfs, metrics-server), rolls talosctl upgrade one node at a time, re-applies the imperatively-installed services (MetalLB / Argo CD / Sealed Secrets), then hard-refreshes every Argo CD Application and restarts the :latest app deployments.

Longhorn is the exception: it is Argo-managed via apps/longhorn.yaml, so its bump flows through Git, not a re-apply - otherwise the app-of-apps root (tracking main with selfHeal) would revert the local change and cause a version flip-flop. When the script finishes, review git diff secrets/cluster.yaml apps/ and commit & push so Argo CD syncs the chart bumps.

Reconfigure or add nodes - the reconfig phase

Once the cluster is running, scripts/12-reconfig.sh pushes a cluster.yaml edit to the nodes without a full rebuild. It regenerates the machine configs (so your edit is always reflected), then decides per node how to reach it: a running node gets an authenticated, live apply - a config-only change such as adding network.dnsServers lands with no reboot (Talos stages one itself only if a change actually needs it) - while a fresh node still in maintenance mode gets the first-boot insecure apply after its maintenance API answers. So you can add nodes to cluster.yaml and run this: the new ones get the first-boot apply, the existing ones reconfigure instantly.

Tear it down

To wipe the cluster's VMs - a botched bootstrap, or reclaiming the hosts - the standalone deprovision phase stops and destroys every VM in your config (disks included) over SSH. It is not part of ./bootstrap.sh and makes you type the cluster name first; VMs that don't exist are skipped, so it's safe to re-run.

DHCP reservations and the Talos ISO are left in place, so a fresh ./bootstrap.sh rebuilds straight over the top. Because qm destroy purges the disks, no Talos state survives on the VMs - etcd, Longhorn volumes and OS are all gone - and the bootstrap keeps talosctl/kubectl scoped to secrets/out (it never touches ~/.talos or ~/.kube). 16Troubleshooting

When a phase stalls

Grouped by where a run tends to stall. Jump to your area, then find your symptom.

Provisioning, boot & DHCP
Stuck at "waiting for maintenance API"

CauseVM didn't get its reserved IP, or didn't boot from the ISO

FixCheck the DHCP reservation matches the VM's MAC; the Talos console shows the node's actual IP

Node boots on the wrong IP (not its reservation)

CauseMAC mismatch, a stale dynamic lease, or dnsmasq isn't the DHCP server

FixRun DHCP_VERIFY=1 ./bootstrap.sh dhcp. It flags each case: set the node's mac: to the VM's real NIC MAC (qm config <vmid> | grep net0); delete a stale lease on OPNsense and reboot; or make dnsmasq the DHCP server on that interface

VM lands in the UEFI/EFI shell instead of booting the ISO

CauseOVMF found nothing bootable - empty disk on first boot, or boot order

FixThe standard metal-amd64.iso is UEFI-bootable; confirm the VM has an efidisk0 and boot order scsi0;ide2. From the shell type exit to reach the boot menu; on first boot the empty disk falls through to the CD

dhcp phase returns HTTP 400/403/404

CauseEndpoint missing, or the API user lacks access

FixThe script prints the status + response body. 400/404 → OPNsense <25.1 or dnsmasq isn't the DHCP server; 403 → give the key's user the Services: Dnsmasq DNS/DHCP privilege; 401 → wrong key/secret; a 400 invalid json syntax means a JSON content-type was sent with an empty body - the shipped script only sets it when there is a body

Config & tooling
yq: command not found

CauseNo yq on PATH

FixInstall either build - brew install yq (mikefarah) or pip install yq (jq-based); the scripts work with both

Storage & the dedicated disk
Node won't boot; "no disks matched for volume"

CauseThe data disk isn't blank, or the selector didn't match

FixWipe the second disk so it's empty, or adjust diskSelector; check talosctl -n IP get volumestatus

apply fails: UserVolumeConfig "min or max size is required"

CauseTalos needs a size bound on a user volume

FixSet longhorn.dedicatedDisk.minSize (e.g. 10GiB, at or below the data disk); the generated volume pairs it with grow: true to fill the disk

Longhorn pods Pending / CrashLoop

CauseExtensions missing, or landed on a tainted control plane

Fixtalosctl -n WORKER_IP get extensions must list iscsi-tools & util-linux-tools

Cluster health, etcd & the VIP
2nd control-plane node never joins; keeps logging "etcd is waiting to join … run talosctl bootstrap"

CauseBootstrap was run on more than one node (split-brain etcd), or the node can't reach the first CP

FixBootstrap runs once, on one node - the message on the others is normal. If you bootstrapped a second node, reset it so it rejoins fresh (talosctl -n IP reset --graceful=false --reboot) then re-apply controlplane.yaml - don't bootstrap. Otherwise confirm the first CP is healthy (talosctl -n 192.168.4.11 get members) and both nodes are on their expected IPs and can reach each other

health fails: rpc error … unable to connect to 192.168.4.12:50000

CausePort 50000 is the node's Talos API (apid); the node is unreachable, not just unhealthy

FixThe node is down, still booting, sitting in the EFI shell, or not at its configured IP. Open its Proxmox console; from the workstation try ping 192.168.4.12 and talosctl -n 192.168.4.12 version --insecure (answers only in maintenance). Confirm the IP with DHCP_VERIFY=1 ./bootstrap.sh dhcp. If you flipped an existing VM from SeaBIOS to OVMF, a disk Talos installed under BIOS won't boot under UEFI - recreate the VM (or wipe and re-apply) so Talos lays down a systemd-boot ESP

After bootstrap, node logs show k8s.EndpointController ... failed to list *v1.EndpointSlice ... no route to host 192.168.4.10:6443

CauseTalos's endpoint watcher can't reach the VIP yet - "no route to host" means the VIP isn't bound on any control plane (etcd not healthy) or its ARP isn't crossing to this node

FixNot fatal: KubePrism filters out the unreachable endpoint and uses each node's local apiserver, so bring-up continues. Confirm the cluster is healthy (talosctl -n 192.168.4.11 -e 192.168.4.11 health) and the VIP is bound (... get addresses | grep 192.168.4.10). Once it's up and reachable across hosts, the message stops

x509: certificate is not valid for 192.168.4.10

CauseThe API server cert doesn't list the VIP

FixMake sure network.apiEndpoint is set before you run config so it lands in the cert SANs (the scripts add the VIP plus every control-plane IP). If you changed apiEndpoint afterwards, regenerate the config and re-apply to the nodes

(talos-vip) only one control plane shows the VIP; the second never claims 192.168.4.10

CauseExpected in talos-vip mode - the VIP is held by one node (the etcd leader), and a node can't claim it until it's a healthy etcd member

FixDon't move it onto the second node; confirm it's on the first (talosctl -n 192.168.4.11 get addresses | grep 192.168.4.10). If the second lacks it, the real issue is that it hasn't joined etcd - fix that and the election follows. If the VIP never appears at all, it isn't bound - check etcd is healthy (it only binds on a healthy leader) and see the cross-host ARP row below

(talos-vip) the VIP answers on the control plane that owns it, but other hosts/nodes get no route to host 192.168.4.10

CauseThe owner's ARP for the VIP isn't crossing the SDN VNet - either a Proxmox NIC firewall (IP/ARP filter) is dropping it, or the VNet doesn't flood ARP between hosts

FixTurn the PVE firewall off on the control-plane NICs (qm set <vmid> --net0 ...,firewall=0), and confirm the VNet floods broadcast across hosts (VLAN/VXLAN flood ARP; a Simple zone is host-local; an EVPN zone needs Disable ARP-ND Suppression or BGP). Verify with arping -c3 192.168.4.10 from another host. Swapping VIP software (e.g. kube-vip) won't help - same ARP path. If the zone can't flood ARP, move to a VLAN or VXLAN zone

Everything on the first host works (cp1 up, its workers Ready) but the second host fails: cp2 never joins etcd and its workers never go Ready

CauseNode-to-node traffic isn't crossing between your PVE hosts. On a VLAN zone the VNet's VLAN is almost certainly not trunked between the hosts on the physical switch - management still works because your router reaches each host's segment, which hides it

FixThis is a network problem, not Talos or the scripts: every node gets the same config and bridge regardless of host, so a clean per-host split is always the cross-host path. Prove it - put a throwaway VM/CT on the lab VNet on each host and ping between them (e.g. .201 ↔ .202); if that fails, cross-host L2 is broken. Fix on the switch/SDN: the VNet's VLAN ID must be allowed on the trunk ports to both hosts, and the zone's underlying bridge must have a physical uplink carrying it. Confirm with bridge vlan show on each host

LoadBalancer IPs, apps & GitOps
Service EXTERNAL-IP stays <pending>

CausePool empty/overlapping, or speaker not ready

FixConfirm the pool is in-subnet, clear of .110–.220, and the speaker DaemonSet is running

Pi-hole DNS doesn't answer on its IP

CauseShared-IP keys differ, or DNS points at the wrong address

FixBoth services need the same allow-shared-ip value; query it with dig @192.168.4.53 example.com

Argo CD can't fetch the repo

CauseDeploy key not added, or an HTTPS URL used with an SSH key

kubeseal can't reach the controller
